Part 1: Closer than you think

The mental shift that turns SharePoint admins into SPFx builders.

  • SPFx
  • Microsoft365

If you can build a SharePoint site, you’re already closer to writing SPFx than you think. Not “with training.” Not “eventually.” Right now.

The gap between “SharePoint admin” and “SPFx developer” is smaller than it looks. Most of it is unfamiliar tooling and a mental model you haven’t seen yet. Once that clicks, the rest follows.

The moment this usually shows up

It starts with something that feels simple. “Can we add a banner to every page?” or “Can this button send documents somewhere automatically?” You try the usual options: built-in web parts, list formatting, Power Automate. At some point, they stop being enough.

That’s where most people stop and say “we need a developer.” That’s the line this series is about crossing.

What SPFx actually is

SPFx (SharePoint Framework) is Microsoft’s supported way to customize SharePoint Online. It’s how you build custom web parts, page-level UI like banners or headers, command buttons inside lists and libraries, and custom forms. All of it runs in the browser, inside SharePoint. No servers, no classic add-ins, no injecting random JavaScript into pages and hoping it keeps working.

The part nobody explains clearly

Every SPFx solution you’ve ever deployed came from the same place: a project on someone’s machine that went through a build step and produced a .sppkg file. That .sppkg file you upload to the app catalog? It’s just the output. Nothing about that process is locked away. You can create that project.

The mental shift

For a long time, I treated SPFx as something developers build, I deploy, and I approve permissions for. That works. But it also means you’re always waiting on someone else for things that often aren’t that complex.

The shift is realizing that if I can build it myself, I understand it better, and I make better decisions about it. That’s the entire goal of this series.

The honest rule

If you can solve something with list formatting, Power Automate, or built-in web parts, do that instead. SPFx is what you reach for when those stop working.

? Unsourced

This is not a formal Microsoft rule. It’s a practical guideline based on how SharePoint solutions tend to evolve in real environments.

What this series covers

I’m not trying to turn you into a full-time developer. The scope is much smaller and more useful: run an SPFx project locally, change one thing, see it update, package it, deploy it. Once you’ve done that once, everything else makes more sense. Permissions, packaging, risk, all of it.

Some parts will feel unfamiliar. You’ll install tools you haven’t used before. You’ll run commands that look a little weird. That’s normal. You don’t need to understand everything immediately. You just need to get something running.
