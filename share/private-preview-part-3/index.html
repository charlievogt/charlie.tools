Part 3: Deploy it

Package your solution, deploy it to SharePoint, and discover the PnP community samples you can build and deploy today.

  • Microsoft365

i Note

The .sppkg file contains metadata about your solution and instructions for SharePoint, but it does not always contain all your code. Sometimes the JavaScript assets are hosted separately (typically on a CDN or an Azure Storage account). Microsoft explains the packaging model here. If something breaks after deployment, the issue may not be inside SharePoint. It could be where your assets are hosted.

Upload to the app catalog

Go to your SharePoint App Catalog and upload the .sppkg file. You’ll be prompted to deploy it.

You may see “Make this solution available to all sites?” If you’re just testing, don’t do that yet. Deploy it to a test site first.

Add it to a page

Go to a SharePoint site, edit a page, and add a web part. You should now see yours in the list. Add it, edit the properties, change the title, save the page. It works.

That’s the full lifecycle: created a project, ran it locally, changed the code, packaged it, deployed it, used it in SharePoint. Before this, SPFx was probably something developers build and you deploy. Now it’s something you can build, something you understand, and something you control. That’s a different position to be in.

One thing to be careful with

When you deploy solutions, you may see permission requests for Microsoft Graph or SharePoint APIs. These are approved separately from the solution deployment itself.

? Unsourced

If you don’t understand why a permission is needed, don’t approve it yet. Ask for a plain-language explanation of what the permission does and why the solution needs it.

You don’t have to build everything from scratch

The tooling you just installed is the same tooling the entire SharePoint developer community uses. And that community has been sharing their work for years.

The PnP (Patterns and Practices) community is a volunteer-driven, Microsoft-supported open-source effort. One of their best resources is the SPFx web part samples gallery: hundreds of ready-to-use web parts, each with source code, documentation, and deployment instructions.

pnp / sp-dev-fx-webparts

Community samples for SharePoint Framework client-side web parts.

↗

These aren’t toy demos. The gallery includes org charts that pull from the Microsoft Graph, FAQ accordions backed by SharePoint lists, file preview web parts, theme managers, and DevOps integrations. Some of them are production-ready as-is. Others are solid starting points you can extend.

The workflow is exactly what you just learned: clone the sample, run npm install, run npm run serve to test it locally, then npm run bundle and npm run package-solution to produce a .sppkg you can deploy. Every command is the same. The only difference is someone else wrote the code.

Browse the gallery and look for something that solves a problem you actually have. An FAQ web part for your HR site, a document card layout for a project hub, an org chart for your intranet. Download it, build it with the tools you already have, and deploy it to a test site. That’s a real solution running in your tenant, and you didn’t have to write it from scratch.

There’s also a separate gallery for SPFx extensions (application customizers, field customizers, command sets) if you want to go beyond web parts.

Where this goes from here

You can keep things simple, or you can build on what you’ve learned. The tools you installed in Part 2 are the same tools I use to build everything I write about on this blog: tenant-wide document scanners, custom announcement systems, intranet search experiences. All of it starts with the same yo @microsoft/sharepoint, the same npm run serve, the same edit-save-refresh loop.

The jump from “HelloWorld with a title property” to “web part that calls the Microsoft Graph” is smaller than it sounds. The jump from there to “web part with React and Fluent UI” is smaller still. Each step adds one new concept to a foundation you already have.
