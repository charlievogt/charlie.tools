Part 2: Build your first web part

Set up your environment, scaffold your first SPFx project with the Heft toolchain, and make your first change.

  • SPFx
  • Microsoft365

If Part 1 was the mindset shift, this is the part where it becomes real. I’m going to walk through installing the tools, creating a project, running it locally, and changing one thing.

Fair warning: most of this happens in a terminal. If you’ve never used one, that’s fine. It’s just a different way to tell your computer to do things: instead of clicking buttons, you type short commands. I’ll explain every one before you run it.

Before you start

You need two things installed before anything else.

Visual Studio Code is a free code editor from Microsoft. It’s not Visual Studio (the big one). It’s lightweight, it has a built-in terminal, and it’s what most SPFx developers use. Download it, install it, open it once to make sure it launches. That’s all you need for now.

Node.js 22 LTS is a runtime that lets JavaScript run on your machine instead of only in a browser. SPFx uses it to build projects locally. Go to nodejs.org, download the LTS version (not “Current”), and run the installer with the default options. Do not use a version other than 22 LTS. SPFx only works with specific Node versions, and mismatches are the single most common reason things fail before you even start. Microsoft documents supported version pairings here.

★ Tip

SPFx 1.22.x requires Node.js 22 LTS. If your Node version doesn’t match your SPFx version, the project may fail to build or behave unpredictably.

SharePoint Framework Toolkit is an optional VS Code extension that wraps Yeoman and heft behind a UI panel. If you’d rather click than type, install it now. The walkthrough below uses the CLI so you can see what each step does. The extension just hides those steps behind buttons.

Open a terminal

A terminal is a text-based window where you type commands and your computer runs them. You type something, press Enter, and something happens. That’s the whole interaction model.

Open VS Code. Press Ctrl + backtick (the key right above Tab; VS Code keeps this one as Ctrl on Mac too). A panel opens at the bottom of the window. You should see a blinking cursor next to some text that looks something like PS C:\Users\yourname> yourname@MacBook ~ % yourname@host:~$ . That text is called the prompt, and it tells you where on your computer you’re currently “standing.” The blinking cursor means it’s waiting for you to type something.

Trust the development certificate

This is the first place things usually go wrong, and it’s a quick fix.

SPFx serves your project locally over HTTPS, which requires a trusted security certificate on your machine. Without it, your browser will block the page. Run this:

npx heft trust-dev-cert
npx heft trust-dev-cert
npx heft trust-dev-cert  # then import the printed .pem into your browser

npx is a companion to npm that runs a tool directly without installing it globally. This command generates a development certificate and asks your operating system to trust it.

A Windows Security dialog appears asking if you want to install the certificate. Click Yes. A Keychain Access prompt asks for your admin password. Enter it to add the cert to your login keychain and mark it trusted. Linux doesn’t have a unified system trust store, so this command can only do half the job. Note the path it prints (typically ~/.rushstack/rushstack-serve.pem) and import that file into your browser as a trusted Certificate Authority. In Chrome: Settings → Privacy and security → Security → Manage certificates → Authorities → Import.

If you skip this step, the browser will show a certificate error later and nothing will load.

Run the project

Start the local development server:

A lot of text scrolls by. This is the build tool (Heft, which replaced the older Gulp-based toolchain in SPFx 1.22) compiling your project and starting a local web server.

When it finishes, your browser may auto-open to a SharePoint URL that doesn’t load. That’s expected. The toolchain defaults to the hosted workbench, which runs your local web part inside real SharePoint chrome and needs your tenant configured first. Close that tab. You don’t need it for this part.

Instead, open the local workbench at https://localhost:4321/temp/workbench.html. The local workbench is a standalone test page that lives entirely in your browser. It isn’t a real SharePoint site, so nothing you do touches your tenant. It’s the right place for “does my web part render?” and it doesn’t need any tenant config.

You should see a mostly blank page with a + button. Click it, you’ll see HelloWorld in the list, and clicking it adds the web part to the page.

★ Tip

If the page won’t load at all (browser shows a connection error or a certificate warning), the build probably failed at startup. Switch to VS Code and look at the terminal for red error text. The two usual causes are the dev certificate from the previous step still not being trusted (re-run npx heft trust-dev-cert) or another process already using port 4321.

★ Tip

If the workbench loads but HelloWorld isn’t in the picker, the dev server is probably still compiling on its first run. Wait until the terminal shows a “compiled successfully” line, then refresh the page.

If it renders on the page, you’re in.

If something broke

Three things cause almost every first-run failure.

Node version mismatch. Run node --version and confirm you’re on 22.x. If you installed a different version, uninstall it and grab 22 LTS from nodejs.org.

Dev certificate not trusted. Run npx heft trust-dev-cert again and make sure you approve the system prompt.

Dependencies not installed. If you see errors about missing modules, run npm install again. Make sure you’re inside the project folder when you do (your prompt should show spfx-first-project).

Start with those three. It’s almost always one of them.
